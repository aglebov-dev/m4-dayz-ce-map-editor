# План: редактор карты CE

Статусы: `[ ]` / `[~]` / `[R]` / `[x]`. Модули: `core/areaflags`, `writer`,
`ce_project`, `diff`, `stats`, `export`; панели слоёв/объектов/инспекторов.

**Цель:** редактор карты CE. Работа с `areaflags.map`; открытие проекта BI
(продолжить начатое); экспорт в проект BI. Опционально — файлы экономики.
Инструменты: Слои, Объекты, Инспекторы, Диф (2 режима). Позже — распаковка спутника.

Провайдеры: `light/providers.py` — Local (ФС) и SFTP (корень сервера DayZ); материализация
в кэш, добавление провайдера ридеры не трогает.

## L0 — Модель проекта + служебная папка `[R]`
Проект = набор путей (map+флаги обязательны; cfglimits; экономика; тайлы; папка проекта BI)
+ ссылка на снапшот. Служебная папка приложения хранит конфиги проектов и снапшоты.

## L1 — Окно открытия = конфигурация проекта `[R]`
Своя форма: выбор конкретных файлов (видно, что именно загрузится), проверка наличия.
Режимы: новый проект / продолжить существующий. Источник — сервер или проект BI.

## L2 — Снапшот для Дифа `[R]`
Новый проект → сохранить первоначальный снапшот данных в служебную папку (эталон сравнения).
В конфигурации: «удалить снапшот» (предупреждение: Диф перестанет работать),
«загрузить снапшот вручную» из проекта BI или из файлов сервера.

## L3 — Блокировка инструментов по наличию файлов `[R]`
Каждый инструмент проверяет свои файлы; нет — задизейблен + подсказка «не загружено: …».

## L4 — Диф, два режима `[R]` (режим A — снапшот; B — проект BI через ce_project)
(а) текущее ↔ снапшот (файлы сервера); (б) текущее ↔ слои проекта BI.
База — `core/diff` + `ce_project` (сверка по именам флагов, не битам).

## L5 — Экспорт в проект BI `[~]` (папка выбирается; TGA-слои — после выбора А/Б)
Своя форма: выбор папки сохранения, список файлов, которые будут записаны.
(Вариант слоёв А/Б — решить перед реализацией записи.)

## L6 — Перенос инструментов `[R]` (панели переиспользованы из родителя, под блокировками)
Слои, Объекты, Инспекторы (+ вода из проекта BI). Редактирование (кисть/фигуры) — по
необходимости, под теми же блокировками.

## L7 — Подложка (спутник) `[R]`
Секция «Подложка» в окне проекта: выбор из хранилища (служебная папка) или изображения;
размер мира берётся из areaflags (масштабирование автоматическое — вопрос «км» снят).
`light/tiles_store.py` (пирамиды в служебной папке + внешние корни M4_TILES_DIRS),
`light/tiles_unpack.py` — ХУК распаковщика (M4_UNPACK_SCRIPT: unpack(game_dir, out, world)).
Распаковка НАТИВНАЯ на Python (light/sat_extract.py — порт ExtractSatMap.cs: PBO→PAA→
DXT1/DXT5 векторно через numpy, LZO1X, склейка, пирамида через Pillow). Без dotnet.
Проверено сквозняком: Chernarus за 21с, meta идентичен C#, карта без швов.

## L8 — Редактирование флагов `[R]`
Пустые слои (0 ячеек) — включаемы, на них можно рисовать. Добавление новых usage/value
(`core/flags.py`: ёмкость usage≤32, value≤8, ниббл→байт при 5-м value; запись
cfglimitsdefinition.xml). Кнопки «＋usage/＋value» в окне; L5-кнопка помечена
предупреждением о незавершённом экспорте TGA-слоёв.

## Рефакторинг — убрать переменные окружения `[~]`
Работа с ФС приложения сведена в `core/paths.py::AppPaths` (синглтон `paths`).
`AppPaths.appdata` переопределяется через env `M4_HOME`; смоук-тесты — через
`M4_HOME` / `M4_SMOKE_SERVER_ROOT`. Env как канал конфигурации — временное решение.
Осталось:
- перестать читать `os.environ` в `AppPaths.appdata` — передавать `appdata` явно
  (конструктор уже принимает; выбрать место, где создаётся единый `paths`);
- смоук-тесты — принимать пути через аргументы CLI или локальный (гит-игнорируемый) конфиг;
- после переноса удалить чтения `os.environ` и упоминания env-переменных из кода и доков.
