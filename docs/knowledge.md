## areaflags.map v1 (ПИШЕМ ТОЛЬКО ЕГО)

```
заголовок 24 байта = 6 × uint32 LE: gridX(4096), gridY(4096), sizeX_м(15360), sizeY_м, 32, 0
слой A (usage): offset 24, gridX*gridY × uint32 — битмаска
слой B (тиры): offset 24 + gridX*gridY*4:
   5 valueflags (Chernarus/Sakhal) → uint8/ячейку
   4 valueflags (Livonia)          → ниббл (2 ячейки/байт, младший ниббл = чётная ячейка)
размер v1 Chernarus = 83 886 104
индекс: idx = row*gridX + col; col = floor(x/cell), row = floor(z/cell); cell = sizeX/gridX
row 0 = ЮГ. Ячейка может нести НЕСКОЛЬКО тиров (битмаска).
```

Порядок битов = позиция флага в `cfglimitsdefinition.xml` карты (с нуля). У Чернаруси бит 16 =
Historical, у Livonia/Sakhal бит 15 = Special, 17 = Underground. Читать без cfglimits нельзя.

## «v2» (+5851 байт; боевой сервер) — РАЗГАДАН: это CRLF-порча, чинится ТОЧНО

**Это не формат.** Файл — обычный v1, прогнанный через текстовую конвертацию **unix2dos**:
перед каждым байтом `0x0A` вставлен `0x0D` (у боевой Чернаруси в слое B ровно 5 851 байт `0x0A` =
значение тира Tier2|Tier4 — отсюда «+5851» и ложная корреляция вставок с границами тир-зон).
Слой A «был идентичен v1» лишь потому, что маска usage `0x0A` (Police|Firefighter) на карте
не встречается. Починка: убрать каждый `0x0D`, стоящий непосредственно перед `0x0A`
(обратимо точно, включая исходные пары 13-10: unix2dos дал бы из них 13-13-10).
Проверка: размер после починки должен совпасть с ожидаемым v1. Реализовано в
`src/core/areaflags.py::_dos2unix`; доказательство — счётчики тиров и Unique побайтово
сошлись с ванилью вне правок сервера, никаких «полос искажений» не существует.

⚠ Следствие для боевого сервера: движок читает испорченный файл как есть — тир-границы в игре
могут быть смещены (~1 строка + дрейф в средней части карты). Стоит проверить диагом и
положить на сервер починенный файл. Устаревшие обходы прототипа (сшивка по строке 3087)
больше не нужны.

Источник порчи ВОСПРОИЗВЕДЁН (2026-07-17): пайплайн сборки на машине владельца — свежая
компиляция проекта CE Tool в `D:\dayz-server\CE\v2` дала файл с той же порчей (+5851, ровно
те же пары 13-10). Порча незаметна, если в слое B нет байтов 0x0A (ячеек Tier2|Tier4) — так
было до майской закраски. После КАЖДОЙ сборки проверять размер / гонять tools/fix_crlf_map.py.
Конкретный шаг-виновник (сам CE Tool или обвязка) пока не изолирован.

## Тайлы подложки (схема CRM)

`assets/tiles/<world>/{z}/{x}_{y}.jpg`, meta.json: tileSize 256, maxZoom 6, pixelsPerMeter 1,
margin 16, width/height = worldSize + 2*margin. Пиксель полного зума: px = margin + x,
py = margin + (worldSize − z). y растёт ВНИЗ (север сверху). Зум z: масштаб 1/2^(6−z).

## Формула спавна CE (доказана экспериментами на диаге)

```
usage_инстанса = usage_группы (mapgroupproto) ∪ площадной usage ячейки (areaflags)
value_инстанса = тир ячейки (areaflags) ∪ <value> группы (mapgroupproto)
предмет пригоден точке: category ∈ контейнер ∧ tag ∧ (usage&) ∧ (value& — если value=0, не ограничен)
экономика: nominal/min/restock из types.xml; cost = приоритет очереди; счётчик по count_in_* флагам
```

Алиасы флагов (`<usage user="...">`) — из `cfglimitsdefinitionuser.xml`.
types-файлы: `db/types.xml` + все `<ce folder=...><file type="types">` из `cfgeconomycore.xml`.

## Грабли (каждая стоила нам часов)

1. TGA слоёв CE Tool: RLE-пакеты пересекают границы строк — PIL падает, декодер свой (tools/af.py).
2. Компилятор BI кладёт usage на 1 колонку западнее нарисованного; тиры — без сдвига.
3. PowerShell для бинарщины непригоден (регистронезависимые переменные) — только Python/numpy.
4. `mapgrouppos.xml` содержит ВСЕ объекты карты (деревья!) — лутабельно только то, что есть в proto.
5. Кириллица: py-файлы и вывод — UTF-8 (`PYTHONIOENCODING=utf-8`), в PS-скриптах — BOM.
6. Редактор и боевая миссия могут молча расходиться — не доверять «должно совпадать».
